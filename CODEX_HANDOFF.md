## Backend work still needed for the requested social UI

See `PHASE_23_SOCIAL_UI_BACKEND_HANDOFF.md` for the typed contract. In short:

1. A privacy-safe public friend profile read with avatar, selected unlocked border, opt-in stats/achievements, mutual persistent rooms, friendship/block state, and Message/Invite permissions.
2. A blocked-user listing so the UI can present and safely execute unblock. `blockUser(id)`/`unblockUser(id)` alone cannot provide a complete management screen.
3. Safe conversation-member presentation fields (display name, avatar, selected border). The current group panel must fall back to a shortened UUID for non-friends.
4. An explicit accepted-friend-to-persistent-party invite/notification contract if a one-click Invite action is desired. Never disclose private room codes through presence.

Do not work around missing server data by deriving it from filtered directory/search results.
